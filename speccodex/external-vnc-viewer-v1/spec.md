# External VNC Viewer v1

## 背景

`Sanaka` 现在已经能把“自己启动的虚拟机画面”通过内建控制台显示出来。

但当前它还不能承担一个更直接、也更实用的角色：

- 用户手里已经有一台远程机器或远程虚拟机
- 那台机器已经开放了 `VNC`
- 用户只想输入一个 `IP` 和端口
- 然后直接在 `Sanaka` 里连接过去看画面

这意味着当前 `Sanaka` 还只能管理“自己的运行时”，不能顺手承担“通用 VNC Viewer”的职责。

这次要补上的，就是这个能力。

## 为什么要做

这个能力值得做，不是为了“多一个协议开关”，而是因为它能让 `Sanaka` 更像一个真正能工作的桌面/网页工具。

主要价值如下：

### 1. 让 Sanaka 不只服务自己的 Runtime

当前内建控制台主要服务：

- `Sanaka` 自己创建的机器
- `Sanaka` 自己启动的 QEMU

但用户现实里经常还会遇到：

- 局域网里另一台机器的 VNC
- 另一台宿主机上的 QEMU / UTM / ESXi 导出的 VNC
- 临时测试环境
- 朋友或另一台电脑上已经运行好的虚拟机

如果 `Sanaka` 能直接连这些地址，它的使用价值会明显上升。

### 2. 统一桌面版和网页版的远程查看入口

现在项目已经同时存在：

- 桌面版 `Sanaka`
- 网页版 `Sanaka Web Mode`

如果外部 `VNC` 连接能力只做在一边，体验会分裂。

这次要把产品语义统一成：

- 桌面版可以从“更多”里连接外部 VNC
- 网页版也可以从“更多”里连接外部 VNC

这样用户不需要重新理解两套产品。

### 3. 保持主界面克制

这个能力是有价值的，但它不是首页主任务。

因此它不应该：

- 变成首页一个大按钮
- 被塞进设置页
- 在主工作区抢主层级

把它放进现有“更多”菜单，正好符合当前产品方向：

- 主界面继续克制
- 工具型动作集中收纳
- 不把首页做成 dashboard

### 4. 让 Sanaka 具备 VNC Viewer 角色

这一点很重要。

这次不是“给当前虚拟机多一个连接按钮”，而是正式让 `Sanaka` 具备：

- 输入地址
- 发起外部 VNC 连接
- 显示远程画面

的能力。

也就是说，产品语义上它开始兼具：

1. 虚拟机管理器
2. 内建控制台
3. 通用 VNC Viewer

## 目标

`v1` 目标是让 `Sanaka` 的桌面版和网页版都能通过“更多”菜单发起一次外部 `VNC` 连接。

最小可用流程如下：

1. 用户打开 `Sanaka`
2. 点击右下侧栏工具区的 `...` 更多菜单
3. 点击 `连接到 VNC`
4. 输入目标地址
5. `Sanaka` 打开一个 Viewer 页面并尝试连接
6. 连接成功后显示远程 VNC 画面

## 非目标

这次明确不做：

- 不做 `SPICE` 外部连接
- 不做 `RDP`
- 不做地址收藏夹
- 不做最近连接历史
- 不做多连接标签页管理
- 不做文件传输
- 不做剪贴板同步增强
- 不做认证网关
- 不做加密隧道配置
- 不做 SSH tunnel 向导
- 不做企业级连接配置页

`v1` 就只解决：

- 用户输入一个 `VNC` 地址
- `Sanaka` 把它连起来

## 产品原则

### 1. 入口必须放在现有“更多”菜单

这次新增能力必须挂在你截图里的现有 `...` 菜单中。

也就是说它属于：

- 网页模式
- 外部连接
- 工具型动作

这一类收纳入口。

不允许：

- 首页新增一个“连接 VNC”主按钮
- 设置页新增“立即连接 VNC”
- 顶栏再并排加一个独立高优先级按钮

### 2. 桌面版和网页版保持同一产品语义

无论用户当前在：

- Electron 桌面版
- Web Mode 网页版

看到的入口都应尽量一致：

- 同样叫 `连接到 VNC`
- 同样从“更多”进入
- 同样先输入连接信息
- 同样进入 Viewer 画面

### 3. 不暴露底层术语给普通用户

用户看到的文案应优先使用：

- `VNC 地址`
- `主机`
- `端口`
- `密码`
- `连接`
- `无法连接`

不要把这些暴露成主要文案：

- `RFB`
- `WebSocket proxy`
- `bridge`
- `target host`
- `upstream`
- `raw socket`

### 4. 这是“连接外部画面”，不是“导入机器”

用户从这个入口得到的是：

- 一个远程画面会话

不是：

- 一个可管理机器对象
- 一个 `.saka` 机器包
- 一个会自动进入最近机器列表的新机器

所以外部 `VNC` 会话默认不进入机器对象体系。

## 入口规范

当前“更多”菜单里已经有网页模式相关能力。

`v1` 新增一项：

- `连接到 VNC`

建议菜单顺序：

1. `打开网页模式`
2. `连接到 VNC`
3. `复制网页地址`（如可用）
4. `查看网页服务信息`（如可用）
5. `停止网页服务`（如可用）

如果当前状态下部分网页模式项不可用，也不影响 `连接到 VNC` 出现。

## 交互模型

### 1. 点击后先打开连接表单

点击 `连接到 VNC` 后，不直接跳页。

应先打开一个轻量连接表单。

`v1` 推荐形态：

- 桌面版：菜单旁的轻量弹层，或小型对话框
- 网页版：同语义的轻量弹层，或小型对话框

不建议首版直接做成独立复杂设置页。

### 2. v1 必填信息

`v1` 至少支持这些字段：

- `主机/IP`
- `端口`
- `密码`（可选）

也允许提供一个更快的单字段模式，例如：

- `VNC 地址`

支持输入：

- `192.168.1.10`
- `192.168.1.10:5901`
- `example.local:5901`

但内部最终仍应解析成：

- host
- port
- password

### 3. 默认端口

当用户只输入主机、不输入端口时：

- 默认端口使用 `5900`

### 4. 连接结果

连接成功后，应进入一个专门的 Viewer 页面，而不是复用“当前机器控制台”路由语义。

原因：

- 这是外部 VNC 会话
- 它没有 `machineId`
- 它不应伪装成一台 `Sanaka` 机器

建议新增独立语义路由，例如：

- `/viewer/vnc`
- `/sessions/vnc`

具体命名可在实现阶段收敛，但原则必须保留：

- 外部 VNC Viewer 与机器控制台分开

## Viewer 页面规则

### 1. 使用现有 noVNC 技术路线

`Sanaka` 当前已经有：

- `noVNC`
- WebSocket 转发
- 控制台页面

`v1` 应尽量复用这条成熟路径，而不是另起一套显示内核。

### 2. 页面语义

Viewer 页面要明确告诉用户：

- 当前连接的是外部 `VNC`
- 当前不是 `Sanaka` 机器对象

建议显示：

- 页面标题：`VNC Viewer`
- 副文案：当前连接的地址
- 连接状态
- 断开按钮

### 3. 不进入最近机器列表

外部 `VNC` 会话：

- 不写入 `recents` 机器列表
- 不生成机器对象
- 不出现在左侧虚拟机列表里

否则会污染“机器对象”和“临时外部连接”边界。

## 后端要求

因为你这次明确说的是“后端实现”，所以 `v1` 必须把后端职责写清楚。

### 1. 桌面版要能代理外部 VNC TCP 连接

Electron 主进程侧需要提供：

- 校验后的连接请求入口
- 到远端 `VNC` 服务的 TCP / WebSocket 转发能力

当前内部 noVNC 代理主要连接的是本机 runtime 暴露的 websocket 端口。

这次需要扩展成也能连接：

- 用户指定的远程 `host:port`

### 2. 网页版也要能连

Web Mode 下不能要求浏览器直接裸连远程 `VNC TCP`。

因此网页模式也应走 `Sanaka` 服务端中转：

1. 前端提交外部 VNC 连接请求
2. Web Mode service / 主进程建立远端代理
3. 网页端 noVNC 连到 `Sanaka` 暴露的 WebSocket 入口

### 3. 会话必须隔离

外部 `VNC` 连接不应混入现有 runtime machine state。

应当有独立的 Viewer session 概念，例如：

- viewerSessionId
- type = `external-vnc`
- targetHost
- targetPort
- createdAt
- status

但这些内部字段不必全部直接暴露给用户。

### 4. 密码处理

`v1` 至少支持把用户输入的 `VNC` 密码传递给 noVNC / 代理层。

但 `v1` 不要求：

- 系统钥匙串保存
- 记住密码
- 自动填充历史密码

## 错误处理

`v1` 至少需要覆盖这些失败场景：

- 主机为空
- 端口非法
- DNS 解析失败
- TCP 连接失败
- 连接超时
- VNC 认证失败
- 远程主动断开

面向用户的文案要尽量简单，例如：

- `请输入 VNC 地址`
- `端口格式不正确`
- `无法连接到目标地址`
- `密码错误或目标拒绝连接`
- `远程连接已断开`

不要把底层 Node / socket 原始报错直接扔给用户主文案。

## 安全边界

`v1` 不做完整安全体系，但要有最基本边界：

- 不允许把任意原始 Node 错误直接回传成用户主文案
- 不允许前端直接拿原始 TCP 能力
- 连接目标必须经过主进程 / Web Mode service 校验
- 应保留后续加入 allowlist、权限提示、连接确认的扩展空间

## 验收标准

完成后至少满足：

1. 桌面版“更多”菜单中出现 `连接到 VNC`
2. 网页版“更多”菜单中也出现 `连接到 VNC`
3. 用户可以输入 `IP/主机 + 端口`
4. 默认端口为 `5900`
5. 可选输入密码
6. 成功后进入独立 Viewer 页面
7. Viewer 页面能显示外部远程画面
8. 外部连接不会进入机器对象列表
9. 失败时显示用户可理解的错误
10. 桌面版和网页版都通过 `Sanaka` 自己的代理链路完成连接

## 后续版本方向

`v2` 可以考虑增加：

- 最近连接历史
- 收藏地址
- 连接别名
- 只读模式
- 全屏快捷入口
- 更完整的认证提示
- 连接质量/延迟提示
- 多个外部 VNC 会话切换
- 局域网共享和权限收敛

但这些都不属于 `v1` 必做范围。
