# Kimi 这次把导出功能搞偏的记录

这次要直接记清楚：问题不只是“丑”，而是 Kimi 在没看清现有代码和接口的情况下就开始写，结果把前后端对接方向整个带偏了。

说难听点，这次不是在做功能，是在编功能。

## Kimi 这次具体干了什么离谱事

1. 后端明明已经有导出 API，Kimi 还能写出一份“需要后端新增导出 API”的文档。
   这不是正常遗漏，这是连现有接口都没先核对就开始输出。

2. 真实接口明明是：
   - `window.electronAPI.machine.exportMachine`
   - `window.electronAPI.machine.cancelExport`
   - `window.electronAPI.machine.onExportProgress`

   Kimi 还能凭空写成：
   - `export.exportMachine`
   - `export.onProgress`

   这不是实现偏一点，这是接口名都现编。

3. `files.openFolder` 明明已经接好了，Kimi 还写“需要后端实现”。
   说明它根本没把现有 preload、types、main 端链路过一遍。

4. 导出弹窗里还在用 `setInterval` 假跑进度，居然还能写“前端已完成全部 UI 和交互”。
   这话写得非常不负责。假进度不是交互完成，顶多算演示壳子。

5. 目录选择居然拿 `pickDisk()` 冒充“选文件夹”。
   这个不是审美问题，是功能逻辑都没站稳。

## 本质问题

这次 Kimi 最大的问题不是水平不够，而是流程不对：

- 没先看真实接口
- 没先核对现有能力
- 没先区分“演示效果”和“真实功能”
- 先把样子做出来，再倒推需求，再反过来要求后端配合假设

这套工作方式很容易把项目带到一种很烦的状态：

- 看起来做了很多
- 实际不能用
- 文档还会误导下一步的人继续返工

## 这次正确的结论

这次不是“后端没做完，所以前端先模拟一下”这么简单。

真实情况是：

- 后端主导出能力已经存在
- 前端没有认真对接
- 文档还把已存在的东西写成不存在

所以这次要改的不是颜色，不是圆角，不是排版先后。
先把脑子接回真实代码和真实接口。

## 后续要求

Kimi 后面再碰这块，至少要做到：

1. 先看 `types`
2. 先看 `preload`
3. 先看现有调用链
4. 确认真实 API 名称和返回结构
5. 不要再把假进度、假目录选择、假完成态说成“已完成交互”

## 一句话总结

这次 Kimi 不是把导出功能做丑了。
是把一个本来能继续收口的功能，做成了“界面像做了，实际全靠编”的状态。
