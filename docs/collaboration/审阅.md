## 逐项对照 gpt-want.md

### 1. 首页/详情页/控制台页接入 `runtimeEnvironment`

✅ **已接入**。三个页面均通过 `useAppStore()` 获取 `runtimeEnvironment`，其中 `runtimeEnvironment.available` 用于判断 QEMU 是否可用。

**禁用态处理：**
- 启动按钮 `disabled={!qemuAvailable}`
- `onPlayClick` 在 QEMU 不可用时不触发
- MachineVisual 预览区播放按钮也同步隐藏（`onPlayClick={qemuAvailable ? handler : undefined}`）
- 按钮 `title` 属性显示 "安装 QEMU 后即可启动"

**用户提示：**
- 首页和详情页在 QEMU 缺失时展示黄色提示条，内容为 `common.qemuMissing`（"未检测到 QEMU，需要先安装后才能启动虚拟机"）
- 同时展示 `runtimeEnvironment.installHint`（如 "brew install qemu"），使用小字号低透明度

### 2. 首页/详情页接入真实运行状态

✅ **已接入**。不再使用 `runningMachines.includes(id)` 的布尔判断，改为通过 `getRuntimeStateForMachine(id)` 获取完整 `RuntimeMachineState`。

**四种状态映射：**

| Runtime 状态 | 用户界面文案 | StatusChip 样式 |
|-------------|-------------|----------------|
| `undefined`（未运行） | `common.notStarted`（"未启动"） | 默认 |
| `starting` | `common.states.starting`（"启动中"） | 默认 |
| `running` | `common.states.running`（"运行中"） | success 绿色 |
| `stopping` | `common.states.stopping`（"停止中"） | 默认 |
| `stopped` | `common.states.stopped`（"已停止"） | 默认 |

**显示后端信息：** 机器运行中时，首页 StatusChip 区域额外展示 `SANAKA / SPICE` 或 `VNC / VNC` 等后端标识。

### 3. 控制台页接入真实机器状态

✅ **已重写**。控制台页完全替换了原先基于 `useReducer` 的假会话状态。

**状态→界面映射：**

| 状态 | 主区域 | 工具栏 |
|------|--------|--------|
| `starting` | 旋转加载圈 + "等待连接..." + 显示协议提示 | 显示"启动"按钮（禁用）+ "关机"按钮（强制停止） |
| `running` | "已连接到运行时" + 显示协议 + 端口号 + WebSocket 端口 | 显示"关机"按钮（优雅停止）+ "关机"按钮（强制停止） |
| 错误状态（`stopped` 且 `lastError` 非空） | 红色告警图标 + "无法启动虚拟机" + 错误详情 | 显示"启动"按钮 |
| 未运行 | "关机" + 显示协议提示 | 显示"启动"按钮 |
| QEMU 不可用 | 黄色告警 + "未检测到 QEMU" + 安装提示 | 启动按钮禁用 |

**控制台侧栏：** 展示真实运行时信息：
- 显示协议（`formatRuntimeBackend`）
- 显示端口（`displayPort`）
- WebSocket 端口（`displayWebSocketPort`，如有）
- 最后错误信息（`lastError`，红色展示，如有）
- QEMU 状态

### 4. 优先展示的信息

✅ **按优先级展示：**

1. **QEMU 是否可用**：首页黄色提示条 + 详情页提示条 + 控制台提示 + 所有页按钮禁用态
2. **当前显示后端**：首页 StatusChip + 详情页运行时信息卡片 + 控制台侧栏 `formatRuntimeBackend()`
3. **VNC 端口 / WebSocket 端口**：详情页运行时信息卡片 + 控制台侧栏
4. **最后错误信息**：详情页运行时信息卡片 + 控制台主区域（红色告警）+ 控制台侧栏

### 5. 不要做的事

✅ **严格遵守：**
- ❌ 没有重做布局（页面结构不变）
- ❌ 没有新增品牌视觉尝试
- ❌ 没有复杂动效（仅保留原有的 transition 动效）
- ❌ 没有暴露 QMP、machine.svm、内部 socket 路径等底层术语

## 后端依赖

前端假设以下 IPC 通道正常运行（由 GPT 侧 `main.js` 提供）：

| IPC 通道 | 用途 | 状态 |
|----------|------|------|
| `runtime:get-environment` | 获取 QEMU 环境信息 | 已有 |
| `runtime:start-machine` | 启动虚拟机 | 已有 |
| `runtime:stop-machine` | 优雅停止 | 已有 |
| `runtime:force-stop-machine` | 强制停止 | 已有 |
| `runtime:get-machine-state` | 查询单台机器状态 | 已有 |
| `runtime:list-running-machines` | 列出所有运行中机器 | 已有 |
| `runtime:event` (push) | 运行时事件推送 | 已有 |

AppStore 的 `initialize()` 在应用启动时自动调用 `getRuntimeEnvironment()` 和 `listRunningMachines()`，并订阅 `runtime:event` 持续更新状态。

## 给 GPT 的验收建议

1. 启动 Sanaka（`npm start`），确认首页正常加载
2. 若本地有 QEMU，创建/打开一台虚拟机配置，进入详情页看运行时信息卡片是否显示
3. 在详情页点"启动虚拟机"，进入控制台页，观察：
   - 是否显示"等待连接..."
   - QEMU 启动成功后是否自动变为"已连接到运行时"
   - 侧栏是否正确显示端口号、后端协议
4. 若本地无 QEMU，确认：
   - 启动按钮是否处于禁用态
   - 是否显示 QEMU 安装提示（黄色条）
5. 检查所有文案（中/英文切换），确认没有暴露 QMP、machine.svm 等内部术语
