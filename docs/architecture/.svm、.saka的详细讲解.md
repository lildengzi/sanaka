# `.svm`、`.saka` 的详细讲解

这份文档专门讲清楚 `Sanaka` 里两个最容易让人混淆的东西：

- `.saka`
- `.svm`

它会分别从：

- 用户视角
- 产品规则视角
- 文件系统视角
- 开发者实现视角

来解释它们分别是什么、为什么这样设计、实际该怎么理解。

## 先说最短结论

一句话版：

- `.saka` 是 `macOS` 上机器包的**外层外观**
- `machine.svm` 是三平台统一的**内部主配置文件**

也就是说：

- `.saka` 更像“机器包长什么样”
- `.svm` 更像“机器包里面真正的核心配置是什么”

如果再说得更直白一点：

- `.saka` 是盒子
- `machine.svm` 是盒子里的说明书

## 一、`.saka` 到底是什么

`.saka` 不是一个“所有平台都统一使用的单文件格式”。

在当前 `Sanaka` 规则里，它主要是：

- `macOS` 上机器包的外层包名后缀

比如：

```text
Windows XP.saka
Ubuntu Test.saka
我的虚拟机.saka
```

这里的 `.saka` 更接近：

- `.app`
- `.utm`
- `.pages`

这类 `macOS` 上的“包外观”。

也就是说，它看起来像一个文件，但本质上其实是一个目录包。

### `.saka` 在 macOS 上的真实样子

例如：

```text
Windows XP.saka/
  machine.svm
  preview.png
  Disks/
    xp.qcow2
```

所以在 `macOS` 上：

- 用户看到的是一个 `.saka`
- 系统层面它其实是一个目录
- `Sanaka` 把它当成一个完整机器对象

## 二、`.svm` 到底是什么

`.svm` 指的是包内固定名称的：

```text
machine.svm
```

它是当前三平台统一的主配置文件。

它有几个关键点：

- 固定命名
- 使用 `TOML`
- 是唯一主配置源
- 保存机器的核心结构化信息

它不是用户自己随便命名的：

- 不是 `XP.svm`
- 不是 `windows.toml`
- 不是 `config.toml`

而是统一固定成：

```text
machine.svm
```

### `machine.svm` 里面放什么

比如会放这些：

- 机器 `id`
- 标题 `title`
- 架构 `arch`
- 机器类型 `machine_type`
- 启动顺序 `boot_order`
- 磁盘列表
- ISO 路径
- 网络设置
- 显示设置
- 高级 `QEMU` 参数

它本质上就是这台虚拟机的“正式配置定义”。

## 三、为什么不是 `.saka` 里面再放 `machine.toml`

当前规则已经固定：

- 主配置文件就叫 `machine.svm`
- 不再额外保留 `machine.toml`

这样做的原因很现实：

1. 对用户来说更统一
- 外面是机器包
- 里面是固定主配置

2. 对程序来说语义更稳定
- 看到机器包就知道找 `machine.svm`
- 不需要再猜 `toml/json/yaml`

3. 对未来扩展更清楚
- `.svm` 可以被明确理解为 `Sanaka Virtual Machine`
- 不只是一个随便的 TOML 文件

## 四、三平台分别怎么理解

### 1. macOS

`macOS` 的机器包外观是：

```text
<MachineName>.saka
```

例如：

```text
Windows XP.saka/
  machine.svm
  preview.png
```

这里 `.saka` 是外层包外观。

### 2. Windows

`Windows` 下外层不强制叫 `.saka`，而是普通目录：

```text
Windows XP/
  machine.svm
  preview.png
```

打开入口通常是：

- 直接打开 `machine.svm`

### 3. Linux

`Linux` 和 Windows 一样，外层通常也是普通目录：

```text
Windows XP/
  machine.svm
  preview.png
```

打开入口同样是：

- `machine.svm`

## 五、为什么三平台不统一都用 `.saka`

因为产品现在的方向不是“形式统一”，而是“语义统一”。

统一的是：

- 机器包是主对象
- 包内主配置叫 `machine.svm`
- 内部结构一致

不强求统一的是：

- 外层外观

原因是：

### macOS 更适合包外观

`macOS` 用户天然更习惯：

- `.app`
- `.utm`
- `.pages`

这种“看起来像文件、其实是包”的对象。

### Windows/Linux 更适合普通目录

在 `Windows/Linux` 上，普通目录更直观，也更少系统层面的奇怪行为。

如果硬套 `.saka` 外观，反而会带来：

- 文件关联复杂
- 用户理解成本上升
- 系统表现不一致

所以当前方案是：

- 外层外观允许按平台不同
- 内部结构必须一致

## 六、用户应该怎么理解这两个东西

对普通用户来说，最好的理解方式是：

- “机器”才是主对象

而不是：

- “某个配置文件”才是主对象

也就是说，用户不需要天天去想：

- 这是 `.saka` 还是 `.svm`
- 这是包还是文件

用户更应该理解成：

- 这是我的虚拟机

### 用户看到的层级应该是

1. 一台机器
2. 机器有名字
3. 机器有预览图
4. 机器有磁盘、ISO、配置

而不是：

1. 这里有个 `machine.svm`
2. 里面还有个 `preview.png`
3. 外面也许是 `.saka`

这类内部实现细节。

## 七、开发者应该怎么理解这两个东西

对开发者来说，必须把这两层分清：

### `.saka`

它主要是：

- 某个平台上的机器包根路径表现形式

### `machine.svm`

它主要是：

- 机器配置的真正读取入口

所以程序内部最好始终围绕这些概念来工作：

- `bundlePath`
- `configPath`
- `previewPath`

而不是只围绕某个文件后缀写逻辑。

## 八、标准机器包结构

### macOS

```text
Windows XP.saka/
  machine.svm
  preview.png
  Disks/
    xp.qcow2
```

### Windows / Linux

```text
Windows XP/
  machine.svm
  preview.png
  Disks/
    xp.qcow2
```

固定规则：

- `machine.svm` 固定命名
- `preview.png` 可缺省
- 磁盘等资源可以在机器包内继续扩展
- 机器唯一标识放在 `machine.svm` 的 `id` 字段里

## 九、打开行为怎么区分

当前规则里，打开行为按平台分流：

### macOS

选择：

- `.saka`

### Windows / Linux

选择：

- `machine.svm`

### 为什么不统一

因为这符合各平台用户直觉：

- `macOS` 习惯打开包对象
- `Windows/Linux` 习惯打开明确入口文件

## 十、保存行为怎么理解

创建机器时，当前主流程不是“先另存为”。

而是：

1. 根据默认机器目录创建机器包
2. 自动生成外层机器目录/包名
3. 自动写入 `machine.svm`

也就是说：

- `.saka` / 机器目录 是落盘后的机器容器
- `machine.svm` 是写进去的主配置

## 十一、旧格式兼容怎么理解

当前规则允许兼容读取旧单文件 `.saka`。

但要明确：

- 旧单文件 `.saka` 只是导入兼容源
- 不再作为主写出格式

也就是说：

- 旧文件可以读
- 新保存应尽量落成新的机器包结构

## 十二、为什么 `.svm` 比直接用 `.toml` 更合适

如果直接把内部文件叫：

```text
machine.toml
```

虽然技术上也能用，但语义不够清楚。

而 `machine.svm` 的好处是：

- 一眼知道这是 `Sanaka` 的机器配置
- 不会和项目里别的普通 TOML 文件混淆
- 更适合作为产品文件体系的一部分

## 十三、最容易误解的几件事

### 误解 1：`.saka` 是统一单文件格式

不是。

当前它主要是：

- `macOS` 上的机器包外观

### 误解 2：`.svm` 是用户随便命名的虚拟机文件

不是。

当前固定规则就是：

```text
machine.svm
```

### 误解 3：程序主对象是某个配置文件

不是。

程序内部应该始终把：

- 机器包

当成主对象。

### 误解 4：Windows/Linux 没有 `.saka` 就不是机器包

不是。

Windows/Linux 只是外层外观不同，内部结构仍然是统一机器包。

## 十四、一个完整例子

假设用户创建了一台机器：

- 名字：`Windows XP`

那么：

### macOS 上可能长这样

```text
~/Documents/Sanaka/Windows XP.saka/
  machine.svm
  preview.png
  Disks/
    xp.qcow2
```

### Windows 上可能长这样

```text
%USERPROFILE%\Documents\Sanaka\Windows XP\
  machine.svm
  preview.png
  Disks\
    xp.qcow2
```

### Linux 上可能长这样

```text
~/Documents/Sanaka/Windows XP/
  machine.svm
  preview.png
  Disks/
    xp.qcow2
```

这三种外观不同，但语义上是同一类对象。

## 十五、最后再用一句人话总结

如果你是用户，就把它理解成：

- `.saka` 是机器包外观
- `machine.svm` 是机器的核心配置
- 真正重要的不是后缀，而是“这是一台机器”

如果你是开发者，就把它理解成：

- 机器包是主对象
- `machine.svm` 是统一主配置入口
- 平台外观可以不同，但内部结构必须稳定一致
