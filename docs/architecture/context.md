## 这是什么

`Sanaka` 目前是一个 **Electron 桌面应用项目**，目标看起来是做一个 **QEMU 虚拟机管理界面**。

从界面文案和表单字段看，它想提供这些能力：

- 创建虚拟机
- 选择操作系统模板
- 配置 QEMU 架构、启动顺序、声卡、加速方式
- 添加最多 4 个磁盘
- 配置网络模式和网卡类型

但这份判断现在已经过时，需要更新：

- 项目已经接入一版真实的 Runtime Layer
- 已经可以检测系统 QEMU 环境
- 已经可以读取机器包并生成 QEMU 启动命令
- 已经可以启动、重置、终止虚拟机
- 已经接入 QMP 基础控制
- 控制台已经走 `VNC + noVNC` 的真实显示链路

也就是说，`Sanaka` 已经不再只是“前端壳子 / 交互草稿”，而是进入了 **前端正式产品化 + 后端运行时逐步落地** 的阶段。

不过这里仍然需要明确一个项目方向约束：

- 后续前端不按“演示原型”路线推进
- 不接受“最小实现先装样子”的页面策略
- 即使后端暂时缺席，前端也按正式产品界面标准来设计和实现

也就是说，`Sanaka` 当前虽然还没有完成全部虚拟化能力闭环，但方向已经不是“继续补一个原型”，而是把它推进成 **有品牌感、有真实运行能力的正式桌面产品**。

## 现阶段判断

这个仓库当前更像：

1. 一个正在成形的桌面端虚拟机管理器项目
2. 前端已经进入正式产品界面阶段
3. 后端 Runtime、配置存储、QEMU 生命周期和显示链路已经开始落地

但在产品方向上，应当把它理解为：

1. 不是临时 demo
2. 不是“做个原型看看”
3. 是一个正在把正式产品界面和真实运行能力往一起收拢的桌面应用项目

如果你要继续开发，下一步通常会是这几块：

- 继续扩充 QEMU 命令映射的覆盖面
- 提升控制台显示稳定性与显示后端语义

- 增加更完整的运行时错误恢复
- 明确三平台安装、路径、文件关联和分发能力

## Runtime 与控制台状态补充

当前控制台与 Runtime 相关的上下文需要明确补充：

- 控制台已经不是假状态机，而是走真实 `QEMU + VNC + noVNC`
- `RuntimeMachineState` 和 `display connection state` 必须分开理解
- `noVNC` 断线不等于虚拟机关机
- `QMP SHUTDOWN` 不能在非主动停止流程中直接解释为 `stopping`
- 真正的 `stopped` 以进程退出为强信号
- 旧的延迟状态清理可能污染新状态，后续所有异步清理都要防止跨轮次删除新运行态

这条约束已经在 [spec.md](/Users/steve372dzudo/sanaka/speccodex/console-vnc-startup-stability/spec.md) 中单独展开，后续改控制台、显示连接、重启恢复时必须一起参考。

## 机器包规则修订

当前机器配置规则已经进一步收紧，后续实现与文档都应以这一版为准：

- 机器主对象不再按“三平台统一单文件 `.saka`”理解
- `macOS` 外层机器包显示为 `<MachineName>.saka`
- `Windows/Linux` 外层机器目录直接使用 `<MachineName>`
- 三平台包内主配置统一为 `machine.svm`
- `machine.svm` 本身就是 `TOML` 主配置，不再额外保留 `machine.toml`
- 机器预览图固定使用包内 `preview.png`，没有时允许缺省
- 旧单文件 `.saka` 只作为导入兼容源，不再作为首选写出格式

创建流程也已经固定为：

- 新建机器默认直接保存到默认机器目录
- 不把“另存为”作为创建主流程
- 只有用户显式执行“另存到其他位置”时，才进入手动选路径流程

默认机器目录的产品规则是：

- 优先使用设置中的 `defaultSaveDirectory`
- 未设置时默认使用用户 `Documents/Sanaka`
- 创建后应立即形成一个可管理的机器对象，而不是停留在未落盘草稿状态

## 设计工作流程约束

后续凡是涉及 UI、视觉、交互、布局、导航、动效、可用性或前端设计质量的改动，开始前必须先阅读相关 Codex design skill，再进入实现。

默认读取顺序：

- 综合界面评审先读 `webdesign-review`
- 布局、层级、留白、组件先读 `ui-design`
- 侧栏、导航、对象列表先读 `navigation-design`
- 字号、字重、文字层级先读 `web-typography`
- 可用性、减少认知负担、空状态和操作路径先读 `usability`

这条流程用于防止后续继续凭感觉加装饰、加按钮、加卡片，或者再次把 Sanaka 做成宣传页、浮动卡片堆和胶囊控件堆。

首页方向已经进一步收紧，不再把首页当成“品牌宣传页”。

这里要明确几个首页约束：

- 首页参考原始 `renderer` 版本的安静、直接、桌面应用感
- 首页必须使用现有小鱼 logo 资源 `assets/icons/logo.svg`
- 首页允许品牌化，但品牌表达要克制，不能靠超大 Hero 和大段宣传语堆满首屏
- 首页重点是进入工作，而不是解释产品理念
- 首页必须保留明显的顶部空白拖动栏，不能把内容直接顶到窗口上沿
- 首页不再套外层整块淡紫色圆角底板；左侧栏是贴边应用结构，不是独立圆角卡片
- 左侧栏层级通过右侧阴影和细分隔线表达，不通过外框、圆角或大面积边框表达
- 左侧栏不能被做成脱离窗口的浮动卡片；它应当贴边、结构化、像桌面应用的侧栏
- 左侧虚拟机列表不应每行都做成重按钮或胶囊卡；选中态用细侧边指示、浅底和状态点即可
- 首页主体需要缩小并居中，四周留出稳定缓冲区，不能再铺满画布
- 首页标题、按钮和说明文案都应收敛，不能再出现超大字号和大面积粗体
- 首页主按钮禁止渐变，使用克制的纯色淡紫表达即可

因此首页的产品角色调整为：

- 应用启动面
- 机器工作台入口
- 虚拟机配置与模板入口
- 设置与 About 的稳定导航壳

首页新的结构原则是：

- 顶部先保留一整行明显的空白拖动栏
- 左侧主栏不再是通用页面导航，而是“机器对象列表 + 底部工具位”
- 左栏顶部只保留品牌与 `创建虚拟机`
- 左栏中段优先展示虚拟机对象本身，机器就是主要导航
- `设置 / 关于` 固定收到底部工具位
- 右侧主工作区优先展示当前机器
- 没有虚拟机时，右侧显示正式的创建入口和文件入口
- 有虚拟机时，右侧默认显示“当前机器预览 + 状态 + 关键信息”，只保留少量核心操作
- 有虚拟机时右侧只保留一个主焦点，不要再叠加“概览卡片 / 重复摘要 / 一排通用按钮”
- 有虚拟机时不要在标题区再堆 `打开 / 导入 / 创建` 这类通用动作按钮
- 其余机器只作为次级紧凑网格存在，不能再让单卡撑满首屏
- 空状态不再放预览占位图，也不再额外包一层边框卡片；正文和主按钮直接居中
- 首页、详情页、创建页、设置页共用同一套左侧导航与外层留白
- 页面主体与窗口边缘之间保留稳定缓冲区，不能贴边撑满
- 左栏与主工作区之间要留出明确呼吸感，不能挤成一整块展示框
- 标题区、主操作区、内容区之间必须有可感知留白，不能层层堆满

首页文案也要继续收紧：

- 用户界面不主动暴露 `.saka` 这个内部文件格式名
- 与打开已有机器相关的入口统一使用“打开虚拟机配置”这类用户向表达
- 用户界面不展示内部路径、内部配置文件名、内部实现说明或开发阶段解释；这些只写在文档里
- 机器状态用用户能理解的状态文案，不要用保存状态冒充运行状态
- 首页和详情页不把未运行状态写成操作词 `关机`；未接入 Runtime 前默认显示 `未启动`，未来可按真实状态显示 `已关闭`
- 控制台可以使用 `关机`，因为这里是用户执行运行控制的位置
- 关机时主操作文案为 `启动虚拟机`，开机后主操作文案为 `进入虚拟机`

当前 React 版首页的问题，不是功能错了，而是气质偏了：

- 内容撑得太满
- 首屏 Hero 过大
- 视觉更像宣传页或产品落地页
- 与原始桌面应用首页的安静感不一致

后续首页实现应持续遵守这一修订：

- 增加留白
- 降低装饰密度
- 保留淡紫色和 Material You 气质
- 保留品牌，但不让品牌叙事压过工作流入口
- 把“对象优先”视为稳定产品方向，不再退回“页面优先首页”
- 虚拟机预览优先使用真实截图；没有截图时只允许克制浅紫占位

这些限制视为长期约束，后续不要再次自行偏离：

- 不允许再引入渐变主按钮
- 不允许再使用超大标题
- 不允许再让粗体大面积泛滥
- 不允许再套外层整块底板
- 不允许再把内容铺得过满
- 不允许把首页做回宣传页、展示框或营销 Hero
- 不允许为了层级感把侧栏拆成独立圆角浮层
- 不允许把机器状态、导航项、弱标签全部做成显眼胶囊
- 不允许在右侧当前机器区域重复展示同一组信息

## 设置与机器选项边界

- 应用设置页不再提供“默认网络模式”
- 网络模式属于单台虚拟机配置，创建页默认使用 `User`，用户按机器自行调整
- 应用设置页不再提供“默认音频方式”
- 音频方式属于单台虚拟机的高级选项，默认值为 `系统自动`
- 普通用户不应在应用设置里被迫理解 PipeWire、PulseAudio、CoreAudio、DirectSound 等平台音频后端
