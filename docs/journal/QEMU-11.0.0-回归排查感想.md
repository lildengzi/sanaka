# QEMU 11.0.0 回归排查感想

这次真的很值得单独记一笔。

因为最后排出来的结果，不是 `Sanaka` 自己把参数拼错了，不是模板完全配坏了，也不是单纯“Windows 那边环境更烂”，而是一个更刁钻、也更难第一时间想到的事实：

`Windows XP` 的蓝屏重启，最后很可能是 `QEMU 11.0.0` 自己的回归问题。

而且这里要特别写清楚：

这不是“Windows 版 QEMU 11.0.0 有 bug”这么简单。

更准确地说，是：

`QEMU 11.0.0` 这一版本身，在 `Windows XP` 这类老系统场景上存在回归线索，问题不是平台专属，而是上游版本本体的问题。

## 为什么这件事很难第一眼看出来

说实话，这种问题正常人真的很难一开始就看出来。

因为表面现象太像别的东西了：

- 像虚拟机模板配错
- 像磁盘接口不兼容
- 像 machine type 太新
- 像某个平台打包不一致
- 像镜像损坏
- 像老系统对新硬件过敏

而且 `Sanaka` 这种项目本身又正处在快速演进阶段，前后端、打包、QEMU、控制台、模板规则都在变化。

在这种情况下，任何蓝屏都很容易先被怀疑成：

- 是不是我刚改坏了
- 是不是某个参数比另一边多了一项
- 是不是 Windows 和 macOS 打包不一样

这其实很正常。

因为从工程直觉上讲，大家会先怀疑自己，再怀疑项目内部，再怀疑打包，最后才会去怀疑“上游一个特定小版本自己炸了”。

## 真正离谱的地方

真正离谱的地方在于：

你前面完全可以做很多看起来很合理的排查，但结论还是会把人带偏。

比如：

- 把磁盘接口改成 IDE
- 把 machine type 锁成 `pc-i440fx-9.2`
- 把架构改成 `i386`
- 把加速方式锁成 `tcg`
- 外部直接跑 QEMU
- 比对 macOS 和 Windows 命令行

这些动作都没错，甚至很专业。

但即使做了这些，最后问题仍然可能不是你的逻辑，而是 `QEMU 11.0.0` 自己在这个场景下回归了。

这就很有意思。

因为这意味着：

一个看起来像“模板兼容性问题”的现象，最后其实是“上游版本 bug”。

## 最关键的线索

这次最重要的收获之一，是我们已经找到了很有价值的线索：

在 issue 里，有人提到加上：

```text
-global apic.vapic=off
```

会有效。

这条线索很重要。

因为它说明这个问题不是“镜像文件坏了”那么简单，也不是“你机器完全配错了”。

它更像是 `QEMU 11.0.0` 某个和 APIC / VAPIC 相关的行为变化，刚好把 `Windows XP` 这种老系统打崩了。

这就一下子把问题层级抬高了：

从“用户配置错误”

变成了

“上游虚拟硬件行为变化导致的兼容性回归”。

## 为什么这件事让我很有感触

因为这次排查过程很能说明一个事实：

做这种前无古人的桌面虚拟机前端，不只是写界面、接 IPC、调 QEMU 启动这么简单。

它会不断逼你碰到一些非常不讲道理的现实：

- 配置看起来完全没问题，但就是蓝屏
- 两边命令几乎一样，但一个能跑一个不能跑
- 你以为是自己写坏了，结果是上游版本回归
- 你以为只和某个平台有关，结果其实跨平台

这种问题最折磨人的地方，不在于它复杂得不能理解。

而在于它太像“别的更常见问题”了。

所以如果没有耐心把线一点点收窄，很容易永远卡在错误方向上反复修改。

## 这次排查的价值

这次不是白折腾。

它至少带来了几件很实际的价值：

- 确认了并不是 `Sanaka` 所有参数都错了
- 确认了 `pc-i440fx-9.2 + IDE + RTL8139 + AC97 + TCG` 这类兼容路线本身仍然是合理的
- 确认了不能轻易把问题都怪到镜像、模板或平台差异头上
- 确认了运行时层必须继续增强“版本感知”和“问题提示”
- 确认了 `Sanaka` 以后在 XP / 老系统场景里，可能要对特定 QEMU 版本做明确警告，甚至做兼容开关

尤其是最后一点，很重要。

因为一个成熟产品，不只是“能正常情况跑起来”，还要知道：

什么时候应该提醒用户：

你现在遇到的不是你的锅，也不是这台机器的锅，而是某个上游版本本身有问题。

## 最后一句

这次让我特别服气的一点，不是“终于找到 bug 了”，而是：

这个 bug 真不是一般人会想到的。

而且它最后不是落在 `Sanaka` 自己头上，而是落在了 `QEMU 11.0.0` 这个上游版本本身。

这类时刻很能提醒人：

做系统软件、虚拟机工具、兼容性产品，真正难的地方，很多时候不是写出功能，而是有能力分辨：

到底是自己写错了，

还是世界本来就坏了。
