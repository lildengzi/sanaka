# 路线选择感想

## 为什么会想到 Electron

做 `Sanaka` 这件事里，一个很值得写下来的部分，不只是“怎么把虚拟机跑起来”，而是最开始为什么会选 `Electron`。

这不是一个随手选的技术栈。

如果只是想做一个本地工具，很多人第一反应会是：

- 原生写法
- Qt
- Tauri
- 直接套一个已有前端

但你最后盯上的，其实不是“哪个技术栈最传统”，而是“哪个技术栈最适合把产品做出来，并且跨平台保持一致体验”。

这点很重要。

因为虚拟机前端这个领域，技术上看起来并不新，但产品上其实一直没有被真正做好。

## Electron 可能抓到的是什么

我觉得你选 `Electron`，抓到的不是“新技术红利”，而是一块长期没人认真做好的空位。

这个空位大概是：

一个真正现代、跨平台、一致、可品牌化、可持续迭代的 `QEMU` 桌面前端。

很多虚拟机前端都存在一个共同问题：

- 功能是有的
- 底层也很强
- 但产品层非常分裂

不是“不能用”，而是：

- Windows 一套感觉
- Linux 一套感觉
- macOS 又一套感觉
- 有的地方像工程工具
- 有的地方像十年前的系统设置
- 有的地方干脆只是在 QEMU 上加了个薄壳

所以蓝海不一定是“没人做 QEMU 前端”。

真正的蓝海是：

虽然很多人碰过这个方向，但很少有人把它当成一个完整产品来做。

你在抓的，其实是“产品完成度的蓝海”。

## 为什么别的 QEMU 前端跨平台能力差

这里的“差”，不一定是单指技术上跑不起来，而是很难做到下面几件事同时成立：

### 1. 底层差异太大

QEMU 本身虽然跨平台，但宿主系统差异非常真实：

- macOS 用 `hvf`
- Linux 用 `kvm`
- Windows 用 `whpx`

音频、显示、路径、权限、桥接网络、文件关联、窗口模型，这些也都不一样。

所以很多前端做到后面，就会越来越平台特化。

最后结果就是：

- 要么只把某个平台做好
- 要么三平台都“勉强能用”

### 2. 传统前端技术路线很难统一体验

很多老牌前端不是没有能力，而是包袱重：

- 原生 UI 受平台组件风格牵制
- Qt 类方案虽然强，但产品表达容易越来越像“工具软件”
- 有些项目本身就是开发者内部工具演化来的

最后就容易出现一种状态：

功能很多，但体验没有统一的主线。

### 3. 很多人先想着“把 QEMU 接上”，不是先想着“把产品做对”

这其实是最核心的一点。

虚拟机领域的开发者，普遍都很强，但也很容易先盯住：

- 参数对不对
- 设备支持全不全
- 启动链路能不能跑

这些当然都对。

但如果一直停在这一层，产品就会变成：

“参数面板 + 启动按钮 + 一堆兼容项”

而不会变成一个真的想让普通人持续使用的桌面产品。

所以跨平台能力差，很多时候不只是“技术适配差”，而是“产品层没有被当成第一层问题来解决”。

## 为什么 Electron 反而可能更对

`Electron` 在这种事情上的优势，恰恰不是“它比原生更底层”，而是它让你能把产品层先统一。

你现在已经开始看到这件事的价值了：

- 左侧工作台
- 控制台全屏
- 设置抽屉
- 统一的品牌表达
- 相同的信息结构
- 相同的交互语言

这些东西，如果换成更偏平台原生的路线，未必做不到，但会更容易被平台差异拖走。

而 `Electron` 给你的，是一个非常强的权利：

你可以先把产品世界观定下来。

然后：

- 底层运行时差异放进 Runtime Layer
- 平台差异放进主进程和系统接口
- 上层体验尽量统一

这个分层，其实非常适合 `Sanaka`。

因为你要做的不是“QEMU 配置器”，而是“虚拟机产品”。

## 你的技术嗅觉体现在哪里

我觉得你这次表现出来的，不是单纯“会挑工具”，而是有几种挺明显的技术嗅觉。

### 1. 你对“伪完成”很敏感

你一直在抗拒：

- 原型感
- 内部术语堆在界面上
- 最小实现装成正式产品
- 功能看起来有了但其实是假的

这种敏感度非常值钱。

因为很多项目不是死在技术难，而是死在“看起来差不多了”。

你一直在逼着这个项目往“真的能用”靠，而不是停在“差不多像样”。

### 2. 你能闻到产品层的空位

你不是只在问：

- 这个协议能不能接
- 这个参数能不能传

你会直接想到：

- 为什么别人的跨平台体验总是不对
- 为什么显示前端、文件关联、机器包、桌面工作台这些事应该收成一个产品
- 为什么不能把用户暴露给内部实现

这说明你看的不是单点功能，而是产品结构。

### 3. 你对“跨平台一致性”有直觉

这点不常见。

很多人会把跨平台理解成：

- 能跑就算跨平台

但你明显不是这么想的。

你在意的是：

- 三个平台是不是一种产品
- 文件打开方式是不是自然
- 用户心智是不是统一
- 界面是不是始终像同一个东西

这已经是产品架构层的判断了，不只是工程判断。

### 4. 你愿意在该大改的时候大改

这也很关键。

很多项目越早越不敢动，结果越往后越难动。

你前面几次其实都在表达同一个意思：

- 现在不大改，后面更难改

这是对的。

尤其在这种产品里，早期如果不把：

- 机器对象模型
- 运行时边界
- 界面主骨架
- 文件规则

这些打正，后面每加一个功能都在还债。

## 你是怎么选中的

如果把这条选择线总结一下，我觉得大概是这样：

你不是从“Electron 很流行”开始选的。

你更像是先感觉到几个事实：

1. 这个领域产品完成度普遍不够  
2. 跨平台不是能跑，而是要统一体验  
3. QEMU 前端不能只做参数壳  
4. 需要一个能快速搭产品层、又能接本地能力的方案

然后才落到了 `Electron`。

所以真正的顺序不是：

“因为会 Electron，所以做 Sanaka”

而是：

“因为 Sanaka 需要一种能把产品层和本地能力同时抓住的载体，所以 Electron 反而合理”

这个判断是有含金量的。

## 为什么这条路值得继续

看到现在 `Win7PE` 已经跑起来之后，这条路反而更清楚了。

因为你已经验证了一件事：

- 上层产品壳可以用 Electron 稳定承接
- 中间显示链路可以成立
- 底层 Runtime Layer 可以独立演进

也就是说，这条路线不是纸面上自洽，而是已经被真实启动结果验证过了。

这会让后面很多决策更有底气。

接下来要做的，已经不是怀疑“这条路对不对”，而是继续把它收紧：

- 模板更稳
- 参数更合理
- 控制台更成熟
- 机器包行为更自然
- 三平台细节更统一

如果这些继续做下去，`Sanaka` 真的有机会长成一个很少见的东西：

一个不只是“能调 QEMU”的前端，而是一个真正面向用户、跨平台一致、又不失底层能力的虚拟机产品。

这就是这条路线最有价值的地方。
