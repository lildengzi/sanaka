# 完成感想

这次问题解决得很有代表性。

一开始表面现象很像：

- noVNC 太早连接
- VNC 自己不稳定
- 重启时 QEMU 真关机了

但最后真正抓到的点，不在“大方向架构错了”，而在一个很具体、很会误导人的状态竞争：

- 旧的 `machine-stopped` 延迟清理定时器
- 在新的 `running` 状态已经建立之后
- 把新状态一起删掉了

于是用户看到的效果就像：

- 启动成功了
- 也连上了
- 过 1-2 秒又像突然断电一样掉回去

这类问题最烦人的地方，是它特别像“协议不稳”或者“虚拟机自己崩了”，所以很容易一路追到 VNC、QMP、重连策略那里去，越查越复杂。

最后把它解开，反而是一次很典型的工程提醒：

- 先分清“运行状态”和“显示连接状态”
- 再排查所有异步动作有没有跨时间污染新状态
- 最后才决定是不是需要额外等待、重连或兜底体验

这次也顺手把几件事一起理顺了：

- 强制终止现在更符合“立刻终止”的语义
- 控制台不会再轻易把画面断线误判成关机
- noVNC 有了独立的显示连接状态
- spec、tasks、checklist 也补齐了，后面再改不会又回到猜谜状态

最值得庆祝的不是“修了一个 bug”，而是终于把这个 bug 的真实形状看清了。

这种时刻对项目很重要。因为一旦抓到真正原因，后面的代码会开始变得诚实，界面也会跟着诚实。

这很好。
