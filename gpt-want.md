## 当前问题

Sanaka 现在的高级参数还是“一个大文本框 + 尾部追加”的心智。

这会带来几个问题：

- UI 字段和高级参数完全脱节
- 用户看不见当前完整命令结构
- 像 `hostfwd` 这类参数会遇到结构化合并问题
- 继续做“智能识别任意参数”会越来越脆

## 目标行为

前端要把高级参数区逐步改成：

- 按行显示
- 一行一个参数项
- 支持 `+`
- 支持 `-`

并遵守一个核心规则：

- UI 已经有的参数，做双向同步
- UI 没有的参数，不新增 UI 字段

也就是：

- UI 改内存，参数行里的 `-m` 也变
- 参数行里的 `-m` 改了，UI 内存也变
- 但某个 UI 没建模的 `-global`，只留在高级参数层

## 你需要做什么

### 1. 前端按“controlled / custom”两层理解参数列表

你前端这轮不要把所有参数都当一类。

要按这两个语义理解：

- `controlled args`
- `custom args`

### 2. UI 已知参数允许双向编辑

例如：

- 内存
- CPU
- 加速器
- 启动顺序
- 网络核心参数

这些前端需要准备成“可受控参数行”的 UI。

### 3. UI 未知参数不生成新控件

比如用户加了一个：

- `-global ...`

前端不要因此长新表单字段。

它就留在高级参数区里。

### 4. 高级参数区不再是单个大文本框

请按 spec 方向收敛成：

- 按行列表
- 可单行编辑
- 可 `+`
- 可 `-`

### 5. 不要再按“万能尾部文本框”理解 qemu_args

不要继续围绕“一个 textarea 里塞完整高级参数”去做增强。

这轮最重要的是把它转成：

- 参数列表视图
- controlled / custom 语义

## 我这边后端主导的边界

这轮我是按后端主导理解在定规则。

你前端先按 spec 收 UI 结构，不要自己再扩展成：

- 任意参数都自动反推 UI
- 任意参数都生成新控件

## 不要做的事

- 不要把 UI 未知参数反推成新 UI 控件
- 不要让 controlled / custom 同义重复
- 不要继续强化旧的大文本框心智
- 不要承诺“任何 QEMU 参数都能双向同步”
